# 09 — Мультиязычная архитектура сайта (EN / UK / RU) — для технического ответственного

**Дата:** 2026-05-26 · **Папка:** `03_technical_seo`
**Языки запуска:** английский (база), украинский, русский (для русскоязычной диаспоры — НЕ рынок РФ).
**Основание:** Google Search Central (Localized Versions, Multi-Regional Sites, x-default), John Mueller, ISO 639-1/3166-1. Источники — в конце файла.

---

## ВЫВОДЫ ПЕРВЫМ ДЕЛОМ (что критично не перепутать)

1. **Код украинского языка — `uk`, а НЕ `ua`.** `ua` — это домен/код страны, в hreflang он невалиден и ломает кластер. Очень частая дорогая ошибка. Русский — `ru`, английский — `en`.
2. **Субдиректории с обязательным префиксом локали:** `/en/`, `/uk/`, `/ru/`; корень `/` → `x-default`. Не субдомены, не `?lang=`.
3. **Переводить ВЕСЬ контент страницы, а не только обвязку.** Частичный перевод = дубль/тонкая страница. И помнить: **Helpful Content — site-wide сигнал**, тонкие локали тянут вниз ВЕСЬ сайт. → **Не запускать локаль, пока её контент не готов целиком.**
4. **Локализация ≠ перевод.** Боли универсальны, но ключевики, примеры, конкурентная рамка и кризисные ресурсы — разные на каждом языке (см. `01_market_and_audience/02,07,08`). Дословный перевод проваливается и по дублям, и по попаданию в запрос.
5. **Для русского — учитывать Яндекс** (поддерживает hreflang как Google) + Bing-сигналы; для всех — `<html lang>` и `content-language`.

## А. URL-СТРУКТУРА

```
example.com/            → x-default (языковой свитчер ИЛИ редирект на /en/)
example.com/en/...      → английский (база)
example.com/uk/...      → украинский   (язык = uk!)
example.com/ru/...      → русский      (диаспора)
```

- **Субдиректории** (не субдомены): консолидируют авторитет домена; Google считает subdir и subdomain эквивалентными, но subdir проще и дешевле.
- **Префикс локали всегда** (`localePrefix: 'always'` в next-intl / аналог). Никогда не прятать локаль из URL — Google нужны уникальные URL на язык.
- **Слэш в конце** — выбрать один формат и держать одинаково в URL, внутренних ссылках, canonical, hreflang, sitemap. («Консистентность — главный технический SEO-фактор», Mueller, 2025.)
- **Никаких IP/Accept-Language авто-редиректов** для краулера: Googlebot ходит с US-IP без Accept-Language. Языковой свитчер — да; принудительный редирект бота — нет (можно мягко предлагать локаль баннером).

## Б. HREFLANG (самое частое место ошибок — 67% внедрений с дефектами)

Правила:
- **Self-referencing:** каждая страница включает саму себя в набор hreflang.
- **Reciprocal:** если A ссылается на B, B обязана ссылаться на A — иначе пара игнорируется.
- **x-default:** обязателен, указывает на fallback (свитчер на `/` или `/en/`).
- **Коды:** `en`, `uk`, `ru` (ISO 639-1). Если позже регионализировать английский — `en-US`, `en-GB`, `en-CA`, `en-AU` (никогда `en-UK`).

Для 3 локалей `<head>`-метод компактен (3–4 строки на страницу). Пример для пары страниц «почему ты не можешь перестать есть»:

```html
<link rel="alternate" hreflang="en" href="https://example.com/en/why-cant-i-stop-eating/" />
<link rel="alternate" hreflang="uk" href="https://example.com/uk/chomu-ya-ne-mozhu-prypynyty-isty/" />
<link rel="alternate" hreflang="ru" href="https://example.com/ru/pochemu-ya-ne-mogu-perestat-est/" />
<link rel="alternate" hreflang="x-default" href="https://example.com/en/why-cant-i-stop-eating/" />
```

(Тот же набор — на ВСЕХ трёх страницах, включая self.) Плюс продублировать в XML-sitemap (`<xhtml:link>`). Если HTML и sitemap разойдутся в URL для одной пары — Google отбросит пару.

**Для Bing/Яндекс дополнительно** на каждой странице:
```html
<html lang="ru">                                  <!-- соответствует локали -->
<meta http-equiv="content-language" content="ru"> <!-- Bing: слабый, но полезный сигнал -->
```

## В. CANONICAL + I18N

- **Self-canonical на каждой локали** (`/ru/page` → canonical `/ru/page`). **Никогда cross-locale** (рус. → англ. убьёт индексацию русской версии).
- **Canonical должен входить в hreflang-набор**, иначе весь hreflang игнорируется. Canonical перебивает hreflang при конфликте.
- Пагинация: у каждой страницы пагинации в каждой локали — свой self-canonical (не канонизировать на стр. 1).

---

## Г. КОНТЕНТ ПО ЛОКАЛЯМ (риск тонких дублей)

- **Переводить весь основной контент**, включая title/description/H1 (Google определяет язык по видимому контенту, не по `lang`). Непереведённые метатеги в чужом языке роняют CTR.
- **Локали считаются дублями, если основной контент не переведён.** Только переведённая обвязка при общем теле = дубль.
- **Не noindex'ить тонкие локали** (жжёт crawl budget) и не cross-canonical (конфликт с hreflang). Правильно — **не создавать локаль, которую не можешь сделать полноценной.**
- **AI-перевод сам по себе не спам** (стойка Google с середины 2025), но масштабный низкокачественный перевод подпадает под scaled-content-abuse. Поскольку вы «пишете на RU → переводите»: качество и локализация перевода (а не факт перевода) — решают. Желательна человеческая редактура EN/UK.
- **Локальные сигналы:** на каждой локали — местная валюта/контакты, местные кризисные ресурсы (EN: National Alliance for Eating Disorders +1-866-662-1235; UA/RU — подобрать верифицированные локальные), ссылки с локальных сайтов.

## Д. SITEMAP, SEARCH CONSOLE, АНАЛИТИКА

- **XML-sitemap** с `xmlns:xhtml`, каждый `<url>` содержит `<xhtml:link>` на все 3 локали (включая себя) + `x-default`; абсолютные URL. Делить sitemap по типу контента, не по локали (иначе reciprocal-ад).
- **Лимиты:** 50 000 `<loc>` / 50 МБ; `<xhtml:link>` в лимит URL не входят. На старте неактуально (страниц мало).
- **Search Console:** добавить отдельные property на субдиректории (`/en/`, `/uk/`, `/ru/`) для пер-локального отчёта (International Targeting report устарел — гео-таргетинг теперь через hreflang + сигналы).
- **Для русской локали** — завести **Яндекс.Вебмастер** (Яндекс поддерживает hreflang) — часть русскоязычной диаспоры использует Яндекс. Аналитика: GA4 для всех + при желании Яндекс.Метрика на `/ru/`.
- **Next.js-нюанс:** `alternates.languages` НЕ добавляет self-`<xhtml:link>` для самого `<loc>` — добавлять язык текущего URL явно.

## Е. ПЛАН ЗАПУСКА (фазно, чтобы не уронить весь сайт)

1. **Фаза 1 — EN целиком.** Английская база как «дом» авторитета: pillar-страницы + боль-статьи (`02_content/04a`), полный E-E-A-T (`04b` раздел Г). Это рынок, где ниша зрелая и наш ресерч применим.
2. **Фаза 2 — RU (диаспора), затем UK** — **только когда контент локали готов целиком и локализован** (не дословный перевод): местные ключевики (`08`/`07`), местные конкуренты-референсы, местные кризисные ресурсы. Технически: включить локаль в hreflang-набор всех страниц, добавить в sitemap, завести property/Яндекс.Вебмастер.
3. **Резерв (новые рынки)** — при расширении на en-GB/en-CA/en-AU или новые языки (см. `01_market_and_audience/06_new_markets_scan.md`) добавлять как новые субдиректории с тем же hreflang-паттерном; для 10+ локалей перенести hreflang преимущественно в sitemap.

**Анти-паттерны (не делать):** запуск пустых/тонких UK/RU «ради галочки»; `ua` вместо `uk`; cross-locale canonical; перевод только меню при английском теле; IP-редирект краулера; разный трейлинг-слэш между hreflang и canonical.

## 📖 Расшифровка терминов

- **hreflang** — HTML-разметка, указывающая Google языковые/региональные версии страницы.
- **Self-referencing** — страница включает саму себя в набор hreflang; **reciprocal** — версии ссылаются друг на друга взаимно (иначе пара игнорируется).
- **x-default** — версия «по умолчанию» для пользователей, чей язык не совпал ни с одной заявленной.
- **Canonical** — указание «главной» версии страницы, чтобы избежать дублей; нельзя ставить cross-locale.
- **Locale (локаль)** — языко-региональная версия сайта; **subdirectory (субдиректория)** — `/en/`, `/uk/`, `/ru/`.
- **ISO 639-1** — стандарт 2-буквенных кодов языка (украинский = `uk`, не `ua`!); **ISO 3166-1** — коды стран.
- **Helpful Content System** — система Google, оценивающая полезность контента на уровне всего сайта (тонкие локали тянут вниз весь домен).
- **Crawl budget** — лимит страниц, которые поисковик обходит; **sitemap / xhtml:link** — карта сайта / её элемент для языковых альтернатив.
- **KDP** — Kindle Direct Publishing (самиздат Amazon). **Яндекс.Вебмастер** — панель Яндекса (важен для русскоязычной локали).
